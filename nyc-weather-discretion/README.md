# 裁量性は天候感応性を決めるか — 同一モード内トライアンギュレーション

ニューヨークの**シェアサイクル（Citi Bike）**を題材に、
**「天候感応性を決めるのは “曝露”（濡れるか）ではなく “裁量性”（やめられる移動か）か」** を検証した時系列回帰プロジェクト。

姉妹プロジェクトの対照的な結果 — [nyc-taxi-demand]（天候効かず・前日ラグが支配）と
[nyc-citibike-demand]（天候効く・天候＞ラグ）— の**差を生む隠れた軸**を、交絡を潰して同定することが狙い。
本リポジトリはその **Phase A（自転車・同一モード内）** を、事前登録（`PLAN.md` §4）に沿って完遂したもの。

---

## 主要な結論（重要）

> **「裁量性 → 雨に弱い」という事前登録の主仮説は、頑健には支持されなかった（部分反証）。**
> 主指標 W=corr(残差, 降水アノマリ) では **3軸中1軸のみ支持**。
> 一方、副指標 ΔR²（全天候の増分予測力）では **3軸すべてでレジャー側が高い**。
> **両指標の食い違いこそが結論**で、裁量性は「雨アノマリ単体」ではなく
> **「気温を含む天候全体への感応」として現れる**（＝レジャー＝“好天に乗る”が主）。

### なぜ「識別」できるのか（設計の核）

天候感応性には3つの軸が絡む: **曝露**（濡れるか）・**裁量性**（やめられるか）・**モード代替**（雨で他モードへ流出入）。
題材間比較（自転車 vs 地下鉄 vs タクシー）は3軸が同時に動き**識別不能**。
そこで本研究は **同一モード（自転車）内で、トリップ種別・時間帯・会員種別だけを動かす**。
これで曝露と代替の向きを（ほぼ）固定したまま、**裁量性だけを単離**する。

さらに、季節交絡を[nyc-citibike-demand]から継承した2点で構造的に排除:
1. 天候は**アノマリ化**（実測 − day-of-year 平年値、平年は train のみで推定）→ 季節と直交。
2. ベースラインに**季節（調和3次）＋曜日＋成長トレンド**を入れる（生 weather は入れない）。

主指標 **W = corr(暦ベースライン残差, 降水アノマリ)** は、スケール不変・非季節でクリーン、
**負ほど雨に弱い**。3つの独立な裁量プロキシで「レジャー側ほど W がより負」が一致するかを見る。

## アブレーション結果（`results/segment_weather_sensitivity.csv`）

train=2022-01〜2023-06 / val=2023 Q3 / test=2023 Q4。W と相関は全期間で算出（薄いセルの安定化）、
ΔR² は test の線形（RidgeCV）増分。

| セグメント | n_days | 件/日 | **W**=corr(降水h) | corr(降水量) | corr(最高気温) | ΔR²(test) |
|---|---|---|---|---|---|---|
| all（全体） | 730 | 88,965 | −0.644 | −0.515 | +0.444 | 0.205 |
| rt_point（片道＝通勤） | 730 | 86,272 | −0.642 | −0.514 | +0.438 | 0.199 |
| rt_round（周回＝レジャー） | 730 | 2,693 | −0.564 | −0.433 | **+0.517** | **0.285** |
| time_wdAM（平日朝＝通勤） | 520 | 15,968 | −0.451 | −0.394 | +0.210 | 0.063 |
| time_weMID（休日昼＝レジャー） | 210 | 30,869 | **−0.667** | −0.649 | +0.488 | **0.579** |
| user_member（会員＝通勤寄り） | 730 | 70,919 | −0.626 | −0.508 | +0.384 | 0.162 |
| user_casual（非会員＝レジャー寄り） | 730 | 18,046 | −0.540 | −0.415 | **+0.485** | **0.262** |

### 3軸の判定（事前登録 §4 に照合）

| 軸 | 通勤側 W | レジャー側 W | 主指標 W | 副指標 ΔR² | 気温相関 |
|---|---|---|---|---|---|
| 軸1 ラウンドトリップ | 片道 −0.642 | 周回 −0.564 | **✗ 反対** | ✓ 0.199<0.285 | ✓ 0.438<0.517 |
| 軸2 時間帯×曜日 | 平日朝 −0.451 | 休日昼 −0.667 | **✓ 支持** | ✓ 0.063≪0.579 | ✓ 0.210<0.488 |
| 軸3 会員種別 | 会員 −0.626 | 非会員 −0.540 | **✗ 反対** | ✓ 0.162<0.262 | ✓ 0.384<0.485 |

- **主指標 W（降水への弱さ）は 1/3 軸のみ支持** → §4 の「軸間で食い違う＝部分反証・考察化」に該当。
- **副指標 ΔR²（全天候の増分）と気温相関は 3/3 軸でレジャー側が高い** → 一貫した二次シグナル。

## 考察 — なぜ主指標と副指標が食い違うのか

事前登録通り、結論は「綺麗な単一軸の反証」ではなく**考察**として述べる。

**(1) 裁量性は「雨アノマリ単体」でなく「天候全体（特に気温）」に現れる。**
主指標 W は降水時間アノマリ**1本だけ**との相関。しかしレジャー系（周回・休日昼・非会員）は
**気温相関が一様に高い（+0.49〜+0.52）**。裁量的な移動は「雨だからやめる」より
「**好天だから乗る**」の寄与が大きく、それは W では捉えられず ΔR²（全天候）でのみ立つ。
→ 裁量性仮説は「雨アバース」に限れば部分反証だが、「**好天プル**を含む天候感応」なら 3/3 で支持。

**(2) 通勤側が「雨に強い」とは限らない — モード代替という交絡。**
軸1・軸3で通勤側 W がむしろより負（雨に弱い）なのは、**自転車通勤者は地下鉄という代替を持つ**ため、
雨の日に自転車→地下鉄へ流出しやすいから、と読める。同一モード内でも「乗るのをやめる（裁量）」と
「他モードへ移す（代替）」は完全には分離できず、通勤側の雨アバースを**代替が水増し**している可能性。
これは設計上の限界であり、PLAN §6 の Phase B（地下鉄との2×2）が本来これを切り分ける。

**(3) ΔR² は暦ベースラインが弱いセルで過大に出る（読みの注意）。**
`time_weMID` の ΔR²=0.579 は突出するが、その暦のみ R²(A)=0.112 と極端に低い
（休日昼レジャーは曜日・季節でほぼ説明できない）。**天候が拾える残差が大きいほど ΔR² は膨らむ**ため、
r2A が大きく異なるセル間の ΔR² 直接比較は割り引いて読む。**この交絡を持たない W を主指標に据えた**のは
まさにこのためで、その W が保守的に「部分反証」と言うのが最も信頼できる読み。

**結論の要旨:** 「裁量性が天候感応性の隠れた軸である」ことは、**天候全体（好天プル＋雨アバース）の
水準では 3 プロキシ一致で支持**される。しかし事前登録した狭い操作化（降水アノマリへの弱さ）では
**支持は 1/3 に留まり、通勤側のモード代替が交絡する**。裁量性は実在するが、
綺麗な単一軸「雨アバース」ではなく、**気温を含む天候感応の総体**として現れる。

## Phase B の結果（自転車 × 地下鉄 2×2・曝露の対照、実施済み）

Phase A の限界（モード代替の交絡で軸1・軸3が使えず、支持は時間帯軸のみ）を受けて、
`PLAN.md` §6 で構想していた Phase B（地下鉄を加えて曝露を対照する 2×2）を実施した。
時間帯軸（平日朝=通勤 vs 休日昼=レジャー）だけは自転車・地下鉄の両方に適用できるため、
この軸に絞って「自転車（曝露あり）×地下鉄（曝露なし）」×「通勤×レジャー」の 2×2 を組んだ。

### 結果（`results/phaseB_2x2_mode_segment.csv`, W=corr(残差, 降水時間アノマリ)）

| | 通勤（time_wdAM） | レジャー（time_weMID） |
|---|---|---|
| 自転車（高曝露） | W = **−0.451** | W = **−0.667** |
| 地下鉄（低曝露） | W = **−0.078** | W = **−0.465** |

- **列効果（裁量性: レジャー−通勤）**: 自転車 −0.216 / 地下鉄 −0.387 → **両モードで同方向**
  （レジャー側がより雨に弱い）。Phase A の時間帯軸の結果（軸2支持）が地下鉄でも再現された。
- **行効果（曝露・モード代替: 地下鉄−自転車、同一時間帯内）**: 通勤 +0.373 / レジャー +0.202 →
  地下鉄は両時間帯とも自転車より雨に弱くない（濡れないので直感的に妥当）。
- |列効果|の最大 (0.387) と |行効果|の最大 (0.373) は**僅差**。

### 読み方（正直な着地）

裁量性（列）の効果と曝露（行）の効果は、どちらも同程度の大きさで観測された。
**どちらか一方が支配的、とは言えない。** 裁量性は自転車・地下鉄どちらの中でも再現されたので
「実在する効果」と言えるが、地下鉄が自転車よりずっと雨に強いという行方向の差も同程度に大きく、
Phase A で懸念した「モード代替が通勤側の雨アバースを水増ししている」可能性を否定できない。
→ **裁量性も曝露も、単独では天候感応性の差を説明しきれない。両方が乗っている、というのが
最も正直な読み。**

### 実施メモ・限界

- **期間はbikeと揃えていない**（bike=2022-2023, subway=2023-06〜2024-08）。MTA利用者数はコロナ
  回復が非線形（2022年+33%, 2023年+15.6%）のため、回復が安定した2023年6月〜2024年8月を選び、
  **非線形スプラインを実装せず既存の線形トレンド項のみで済ませた**（期間を選べば `PLAN.md` 原案の
  「MTA hourly の非線形COVID回復トレンドの改修が必要」という前提は回避できた）。
- 天候アノマリ化は期間差の影響を緩和するが完全には除去しない。W はスケール不変・非季節なので
  比較の物差しとして機能するが、この点は限界として残る。
- 会員種別・往復軸は地下鉄の改札データに対応概念がない（終点情報がない・登録会員制度が異なる）
  ため、Phase Bは時間帯軸のみで比較した。
- `tests/test_no_leakage.py` を4種→6種に拡張（地下鉄取得失敗時の非0埋め検証、bike/subway 間の
  時間窓config共有の検証）、全て通過。

```
nyc-weather-discretion/
├── PLAN.md                       # 事前登録（§0問い/§1識別/§2 Phase A/§4 事前登録表/§6 Phase B・実施済み）
├── config/config.yaml            # 期間・セグメント定義・気象・ベースライン・分割の単一の真実（+ phase_b ブロック）
├── src/
│   ├── ingest_segments.py [A]    # 既存 Citi Bike キャッシュ zip を「セグメント別日次件数」に再集計（再DLなし）
│   ├── ingest_weather.py         # Open-Meteo 過去気象（NYC重心1点）。fetch_weather(period, weather_cfg) をA/Bで共有
│   ├── ingest_weather_phaseB.py [B] # Phase B期間(2023-06〜2024-08)の気象を同じ fetch_weather で取得
│   ├── ingest_subway.py    [B]   # MTA Socrata (wujg-7c2s) からシステム全体・時間帯別利用者数を取得→セグメント化
│   ├── merge.py                  # セグメント×気象を date 結合（run_merge(paths) をA/Bで共有）
│   ├── merge_phaseB.py     [B]   # 地下鉄セグメント×Phase B気象を結合
│   ├── baseline.py        [核]   # 曜日+平滑季節(調和3次)+成長トレンド、気象アノマリ（いずれも train のみ fit）
│   ├── analyze_segments.py       # W=corr(残差,降水アノマリ) と A→B 増分R² をセグメント別に算出・対比（analyze_segment()はBも再利用）
│   └── analyze_phaseB.py   [B]   # 地下鉄セグメントを analyze_segment() で評価し、bike(Phase A結果)と2×2に統合
├── tests/test_no_leakage.py      # リーク/交絡検査 6種（B: 取得失敗の非0埋め・時間窓config共有を追加）
├── data/                         # 中間生成物（.gitignore 対象）
│   ├── segments_daily.parquet    #   [date, segment, n]（long, bike）
│   ├── weather.parquet           #   Phase A気象
│   ├── merged.parquet            #   Phase A結合済み
│   ├── phaseB_subway_segments.parquet  # [date, segment, n]（long, 地下鉄）
│   ├── phaseB_weather.parquet    #   Phase B気象
│   └── phaseB_merged.parquet     #   Phase B結合済み
└── results/
    ├── segment_weather_sensitivity.csv   # Phase A最終成果物（セグメント別 W・相関・ΔR²）
    └── phaseB_2x2_mode_segment.csv        # Phase B最終成果物（mode×segmentの2×2, W・ΔR²）
```

### 設定の要点（config/config.yaml）
- 期間: 2022-01-01〜2023-12-31（730日, all 平均 88,965 件/日）
- セグメント: 3軸 × {通勤側, レジャー側} + 全体（計7）。往復=始点/終点駅の一致、時間帯=平日7-9時/休日11-15時、会員=member/casual。
- 分割: train ≤ 2023-06-30 / val ≤ 2023-09-30 / test = 2023 Q4
- 天候: アノマリ（平年偏差, train のみで climatology を fit）。主指標アノマリ = `anom_precipitation_hours`

```bash
# 依存は姉妹プロジェクト ../nyc-taxi-demand/.venv を流用
#   pandas, numpy, scikit-learn, holidays, pyarrow, requests, PyYAML, pytest
PY=../nyc-taxi-demand/.venv/Scripts/python.exe

# A: セグメント再集計（既存 Citi Bike キャッシュ zip を読むだけ・再DLなし）
$PY -m src.ingest_segments      # data/segments_daily.parquet

$PY -m src.merge                # data/merged.parquet

# 分析（W・相関・増分R² をセグメント別に算出・対比）

# リーク検査
$PY -m pytest tests/test_no_leakage.py -v
```

## 既知の限界・今後の課題

- **モード代替の交絡（最重要・Phase Bで着手・部分的に解明）**: 同一モード内でも「乗るのをやめる（裁量）」と
  「他モードへ移す（代替）」は完全分離できず、通勤側の雨アバースが代替で水増しされ得る。これが軸1・軸3で
  主指標を反対に振らせた最有力容疑だった。→ **PLAN §6 の Phase B（自転車×地下鉄の 2×2）を実施**（詳細は
  上記「Phase B の結果」節）。結果、裁量性の効果（列）と曝露の効果（行）は**同程度の大きさ**で観測され、
  **どちらか一方が支配的とは言えなかった**。モード代替の交絡は否定されなかったが、裁量性が「実在しない」
  わけでもない、という両論併記が最も正直な結論。期間はbikeと揃っておらず（bike=2022-23, subway=2023-24）、
  この点は残る限界。
- **操作化の狭さ**: 主指標を降水アノマリ1本に事前登録したため、レジャーの「好天プル（気温）」を取りこぼした。
  多変量の天候感応（雨＋気温＋風の合成指標）を主指標にする再定義は今後の課題（ただし事後変更は事前登録の趣旨に反するため別プロジェクトとして）。
- **薄いセル**: `time_weMID`（休日のみ≈210日）・`rt_round`（2,693件/日）はノイズが大きい。W は全期間で算出して安定化、ΔR² は補助に留めた。
- **ΔR² の baseline 依存**: 暦のみ R²(A) が低いセルほど ΔR² が膨らむ（§考察(3)）。セル間の ΔR² 直接比較は割り引いて読む。

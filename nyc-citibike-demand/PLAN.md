# PLAN.md — NYC Citi Bike 需要における「ベースライン除去後の天候効果」検証

> 実装計画書（精査反映版）。原案（ユーザー作成）に対し、レビューで指摘した統計的論点を
> 反映して設計を確定したもの。**§9 に原案からの変更点とその理由**を明記している。
> 各 Phase 完了後は `@leak-reviewer` で監査してから次へ進む。

## §0 目的
**「シェアサイクル需要は天候感応性が高いが、過去需要ラグが予測を支配するため天候の寄与が
埋もれる。カレンダー・ベースラインを除去し（残差化）、天候を“アノマリ（平年偏差）”として
与えると、天候の寄与が季節交絡なく顕在化するか」**を実データで定量検証する。
評価軸は精度の最大化ではなく **どの特徴が・なぜ・どれだけ効くか**（標準化係数・増分・
悪天候日評価が主役）。前プロジェクト（タクシー＝天候非感応・全 null）との対比に位置づける。

## §1 データ
- 需要: Citi Bike 公式 S3 トリップ（NYC 年次 zip, 2022/2023）。`started_at`→日次、
  `member_casual` 保持。**日次×システム全合算**（空間は扱わない＝過去の空間ミスマッチ回避）。
- 天候: Open-Meteo Historical（NYC 重心1点, APIキー不要）。
- 期間: 2022-01-01〜2023-12-31（年次zip 2本・天候ショック日の検出力確保）。
- スキーマ: Phase0 実測で 2022/2023 はカラム一致（年差正規化は不要）。

## §2 設計（核）
- **目的変数は需要 y に一本化**（total / member / casual）。残差化は内部技法：暦特徴 `cal_*` が
  ベースライン構造そのものなので、線形モデルに `cal_*` を与えること＝「残差予測＋需要復元」と
  代数的に等価。リンゴ・ミカン比較を回避（§9-2）。
- **ベースライン**（`baseline.py`, train のみ fit）＝ 曜日 + 平滑季節(調和3次) + 成長トレンド。
  weather は一切入れない。
- **天候はアノマリ化**（実測 − day-of-year 平年, train のみ fit）。季節と直交するので、
  ベースラインに季節を入れても天候効果を盗まない（§9-1）。
- 特徴の接頭辞: `cal_*`（暦）/ `anom_*`（天候アノマリ）/ `lag_*`（y のラグ）。
- 条件: **A=cal / B=cal+anom / C=cal+lag / D=cal+anom+lag**。
  - B vs A = 天候アノマリの増分（季節交絡なし）。C vs A = ラグの増分。D = 全部。
  - BASE = 暦ベースラインのみの床。
- モデル: Linear(Ridge) / RandomForest / XGBoost。CV=TimeSeriesSplit、予算統一。

## §3 評価（§0 の軸）
- 需要スケールで MAE/RMSE/R²（val でモデル選択、test は最終のみ）。
- **標準化係数**（`results/standardized_coef.csv`, 条件D・線形）: ラグ vs 天候アノマリ vs 暦の
  力関係。**同一目的変数・同一モデル内でのみ比較**（§9-3）。
- **残差×アノマリ相関**（`results/residual_anomaly.csv`）: 「顕在化」を正直に（係数増大は
  目的変数の分散縮小で起こり得るため、predictionの主張は別途 §3 の増分で判定）。
- **天候ショック日評価**（`results/badweather_eval.csv`）: test を「前日から天候が急変した日
  （降水アノマリの前日差・上位10%）」に絞り、C(ラグのみ) vs D(ラグ+天候) の MAE を比較。
  lag-1 が外す日でこそ天候の真価が出る（§9-5）。
- **member vs casual** を主軸対比（casual=レク=天候駆動の仮説）。

## §4 リーク対策（`tests/test_no_leakage.py`, 6検査・先行作成済み）
A ラグ正shift / B rolling当日除外 / C **天候アノマリの train-only fit** /
D **ベースラインの train-only fit** / E **暦特徴に生weather非混入（交絡防止）** /
F ランダム分割なし。加えて ingest は欠損日でRuntimeError（silent 0埋め禁止, R-1）。

## §5 手順（各Phase後に @leak-reviewer）
P0 土台＋実測（天候感応性 corr(temp)=+0.74 確認済み）→ P1 需要取得・集計 →
P2 天候取得・結合 → P3 ベースライン＋特徴 → P4 分割・学習・評価 → P5 分析・図表 → P6 考察。

## §9 原案からの変更点（精査反映・重要）
1. **天候はアノマリ化し、ベースラインに季節を入れる**（原案§3.2の逆）。原案は「月をベースラインに
   入れない（生weatherが季節を盗まれる）」だったが、weather をアノマリにすれば季節と直交するため、
   季節をベースラインに入れる方が正しい。季節を入れないと残差に季節スイングが残り、生weatherが
   “季節の代理”として説明してしまう交絡が生じる。
2. **目的変数を需要 y に一本化**（原案の BASE/LAG/RES 3軸は目的変数も特徴も同時に動かし交絡）。
   残差化は `cal_*` 特徴として内包し、必ず需要スケールで比較。
3. **標準化係数は同一目的変数・同一モデル内でのみ比較**。木モデルは係数を持たないため
   permutation importance で代替（線形係数とは別物）。
4. **成長トレンド項を必須化**（Citi Bike の経年成長 bias 対策。原案は月のみ警戒し trend 未記載）。
   季節月（天候相関→ベースラインでは調和で表現）と成長トレンド（天候非相関→必須）を区別。
5. **悪天候日でなく「天候ショック日（前日からの急変）」で評価**。天候はクラスタし lag-1 が
   織り込むため、急変日でこそ weather の増分が出る。
6. **member/casual を主軸に格上げ**（最も clean な positive が出やすい）。
7. 期間は 2022-2023（原案は Phase0 後に確定→確定済み）。
8. `@leak-reviewer` の監査は各 Phase 後に実施（原案の規律を継承）。

### 想定される結論の型（null も成果）
全体では「ラグ支配が残差化後も残り（lag-1 は短期自己相関で季節除去では消えない）、天候の
*予測*増分は小さい」一方、**標準化係数で天候が顕在化**し、**casual・天候ショック日では天候が
本当に効く**——という、トートロジーでない切り分けを目指す。null でも「なぜ効かないか」を
特定できれば成果とする。
